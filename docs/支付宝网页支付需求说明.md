# 支付宝网页支付需求说明

本文档说明在网站中使用支付宝收款时，在已安装 `alipay-sdk-python` 并配置商户密钥的前提下，仍需完成的配置、功能与非功能要求，可作为实施与验收依据。

---

## 一、背景与目标

- **背景**：网站内发起支付宝收款，服务端使用 `alipay-sdk-python` 与商户密钥完成签名，浏览器跳转到支付宝收银台完成支付。
- **目标**：用户可在网页完成支付；支付结果以**支付宝服务器异步通知**为准；系统可安全、可追溯地更新订单状态。

---

## 二、前置条件（开放平台）

| 项 | 说明 |
|----|------|
| 应用类型 | 网页支付一般对应「**电脑网站支付**」或「**手机网站支付**」（按终端选型，API 不同）。 |
| APPID | 在支付宝开放平台创建应用后获得。 |
| 密钥 | **应用私钥**（仅放服务端）；**支付宝公钥**（或证书模式下的证书）用于验签。密钥格式需与 SDK 配置一致（RSA2 等）。 |
| 签约产品 | 在控制台为应用开通对应支付产品并完成审核。 |
| 网关地址 | 沙箱与生产不同；开发阶段使用沙箱网关与沙箱账号。 |

---

## 三、用户与系统流程

1. 用户在网站点击「去支付」。
2. **前端**调用**自有后端**「创建支付」接口（传业务订单号、金额、标题等）。
3. **后端**校验订单、生成或关联**商户侧订单号**（`out_trade_no`，唯一），调用支付宝 **统一收单下单并支付页面接口**（电脑站：`alipay.trade.page.pay`）或 **手机网站支付**（`alipay.trade.wap.pay`）。
4. 支付宝 SDK 返回 **可跳转的 URL** 或 **自动提交的 HTML 表单**；**前端** `window.location` 跳转或渲染表单提交，进入支付宝收银台。
5. 用户支付完成后：
   - **同步跳转** `return_url`：仅用于展示「支付成功/处理中」页面，**不得**仅依赖此处更新订单。
   - **异步通知** `notify_url`：支付宝 POST 订单结果到商户服务器；**以验签通过后的通知为准**更新订单为「已支付」等。

---

## 四、功能需求

### 4.1 后端

- **配置管理**：APPID、应用私钥、支付宝公钥（或证书路径）、网关 `base_url`、签名类型（如 RSA2）从环境变量或安全配置读取，**禁止**把私钥提交到代码仓库。
- **创建支付**：接收业务参数 → 落库/校验订单 → 调支付宝下单接口 → 返回给前端的应是「跳转 URL」或「表单 HTML」（二选一，与前端约定）。
- **异步通知处理**：提供 **HTTPS 公网可访问** 的 `notify_url`；接收 POST 参数；使用 SDK 或官方规则**验签**；验签成功后解析 `trade_status`（如 `TRADE_SUCCESS` / `TRADE_FINISHED`）；**幂等更新**订单（同一 `out_trade_no` 多次通知只生效一次）；向支付宝返回纯文本 `success`（失败则重试，勿返回 500 乱码页）。
- **同步返回页（可选）**：`return_url` 对应页面可做友好提示，并引导用户「返回订单详情」；订单状态仍以异步通知或主动查单为准。
- **查单补偿（建议）**：对长时间未收到通知的订单，定时任务调用 `alipay.trade.query` 对齐状态，避免漏单。
- **关单/退款（按业务）**：若需要超时未支付自动关单或售后，再增加 `alipay.trade.close` / 退款相关接口与需求。

### 4.2 前端

- 支付按钮触发调用后端创建支付接口，拿到 URL 或表单后完成跳转。
- 支付结果页文案明确：**以订单详情中的状态为准**（若异步未到，显示处理中）。

### 4.3 订单与数据

- 商户订单号 `out_trade_no` 全局唯一，与内部订单一对一或可映射。
- 记录：请求参数摘要、支付宝 `trade_no`、通知原文或关键字段（脱敏）、验签结果、状态变更时间，便于对账与排错。

---

## 五、非功能需求

- **安全**：私钥仅存服务器；HTTPS；防重放（幂等）；通知接口限流与日志。
- **可用性**：通知接口快速返回 `success`，复杂业务可异步队列处理。
- **合规**：金额、币种、商品描述符合支付宝与监管要求；日志勿打完整密钥。

---

## 六、环境与联调清单

1. 开放平台配置 APPID、密钥、签约产品；确认沙箱/生产网关。
2. 本地或服务器部署后端框架（如 FastAPI/Flask/Django），实现「创建支付」+「异步通知」两个路由。
3. 使用内网穿透（如 ngrok）或部署到公网，使 `notify_url` 可被支付宝访问。
4. 用沙箱买家账号完成一笔支付，确认通知到达且订单状态正确。
5. 上线前切换生产网关与正式密钥，复查 `return_url` / `notify_url` 域名与 HTTPS。

---

## 七、验收标准（示例）

- 沙箱环境可完整走通：下单 → 跳转 → 支付 → 收到异步通知 → 订单变为已支付，且重复通知不重复记账。
- 验签失败不更新订单，日志可查。
- 私钥不出现在仓库与前端网络请求中。

---

## 八、与 `alipay-sdk-python` 的关系

- SDK 负责：**组装请求、签名**；解析/验签异步通知（按选用的 API 封装方式调用）。
- 仍需自研：**HTTP 服务、订单库、notify 业务逻辑、前端跳转**，以及 **公网 URL** 配置。
